Controlling the Quadcopter¶

We provide a sample agent in the code cell below to show you how to use the sim to control the quadcopter. This agent is even simpler than the sample agent that you'll examine (in agents/policy_search.py) later in this notebook!

The agent controls the quadcopter by setting the revolutions per second on each of its four rotors. The provided agent in the Basic_Agent class below always selects a random action for each of the four rotors. These four speeds are returned by the act method as a list of four floating-point numbers.

For this project, the agent that you will implement in agents/agent.py will have a far more intelligent method for selecting actions!

import random

class Basic_Agent():
    def __init__(self, task):
        self.task = task
    
    def act(self):
        new_thrust = random.gauss(450., 25.)
        return [new_thrust + random.gauss(0., 1.) for x in range(4)]

import csv
import numpy as np
from task import Task

# Modify the values below to give the quadcopter a different starting position.
runtime = 5.                                     # time limit of the episode
init_pose = np.array([0., 0., 10., 0., 0., 0.])  # initial pose
init_velocities = np.array([0., 0., 0.])         # initial velocities
init_angle_velocities = np.array([0., 0., 0.])   # initial angle velocities
file_output = 'data.txt'                         # file name for saved results

# Setup
task = Task(init_pose, init_velocities, init_angle_velocities, runtime)
agent = Basic_Agent(task)
done = False
labels = ['time', 'x', 'y', 'z', 'phi', 'theta', 'psi', 'x_velocity',
          'y_velocity', 'z_velocity', 'phi_velocity', 'theta_velocity',
          'psi_velocity', 'rotor_speed1', 'rotor_speed2', 'rotor_speed3', 'rotor_speed4']
results = {x : [] for x in labels}

# Run the simulation, and save the results.
with open(file_output, 'w') as csvfile:
    writer = csv.writer(csvfile)
    writer.writerow(labels)
    while True:
        rotor_speeds = agent.act()
        _, _, done = task.step(rotor_speeds)
        to_write = [task.sim.time] + list(task.sim.pose) + list(task.sim.v) + list(task.sim.angular_v) + list(rotor_speeds)
        for ii in range(len(labels)):
            results[labels[ii]].append(to_write[ii])
        writer.writerow(to_write)
        if done:
            break

The Task¶

A sample task has been provided for you in task.py. Open this file in a new window now.

The __init__() method is used to initialize several variables that are needed to specify the task.

  • The simulator is initialized as an instance of the PhysicsSim class (from physics_sim.py).
  • Inspired by the methodology in the original DDPG paper, we make use of action repeats. For each timestep of the agent, we step the simulation action_repeats timesteps. If you are not familiar with action repeats, please read the Results section in the DDPG paper.
  • We set the number of elements in the state vector. For the sample task, we only work with the 6-dimensional pose information. To set the size of the state (state_size), we must take action repeats into account.
  • The environment will always have a 4-dimensional action space, with one entry for each rotor (action_size=4). You can set the minimum (action_low) and maximum (action_high) values of each entry here.
  • The sample task in this provided file is for the agent to reach a target position. We specify that target position as a variable.

The reset() method resets the simulator. The agent should call this method every time the episode ends. You can see an example of this in the code cell below.

The step() method is perhaps the most important. It accepts the agent's choice of action rotor_speeds, which is used to prepare the next state to pass on to the agent. Then, the reward is computed from get_reward(). The episode is considered done if the time limit has been exceeded, or the quadcopter has travelled outside of the bounds of the simulation.

In the next section, you will learn how to test the performance of an agent on this task.

The Agent¶

The sample agent given in agents/policy_search.py uses a very simplistic linear policy to directly compute the action vector as a dot product of the state vector and a matrix of weights. Then, it randomly perturbs the parameters by adding some Gaussian noise, to produce a different policy. Based on the average reward obtained in each episode (score), it keeps track of the best set of parameters found so far, how the score is changing, and accordingly tweaks a scaling factor to widen or tighten the noise.

Run the code cell below to see how the agent performs on the sample task.

Reflections¶

Question 1: Describe the task that you specified in task.py. How did you design the reward function?

Answer: I have specified two tasks, the first task aimed at making the quadcopter stay up for the longest time possible, preferrably around the same area (in terms of x and y) as it started. In order to achieve that, this is how I designed the reward function: (After making modifications to the architecture, I went back and revised the task reward structure to produce best results)

  • I wanted to encourage the quadcopter to go high and keep going up (that will hopefully prevent it from crashing), so I factored in the how many steps have already went ahead in the task into the reward function, multiplying it by the current hight. As a result, the longer the quadcopter stayed up, the higher its reward.
  • As a secondary reward, I didn't want the quadcopter to go very far, so I reduced the reward by a factor of the distance from each of the points to the initial position. A future enhancement would beo play with this factor to see if we can make the quadcopter go to a specific target.

The second task is aimed at starting the quadcopter in the middle of the air, and challenging it to maintain its position as best as it could. The reward function specifies an acceptable error, and if any of the dimensions is within the acceptable error, it gives them a reward. If not, it punishes them by a factor of their distaince from the starting point. Additionally, it provides some punishment for staying below a certain minimum threshold, since we want the quadcopter to keep flying.

An improvement to this task would be to use the euler velocities in order to ensure the quadcopter does not make sudden movements that could cause it to crash, but this will require more knowledge of the physics involved.

Question 2: Discuss your agent briefly, using the following questions as a guide:

  • What learning algorithm(s) did you try? What worked best for you?
  • What was your final choice of hyperparameters (such as $\alpha$, $\gamma$, $\epsilon$, etc.)?
  • What neural network architecture did you use (if any)? Specify layers, sizes, activation functions, etc.

Answer:

  • Since this problem has a continuous action space, the best options are policy gradients and actor critic methods. I've opted to use the Actor Critic method using the starter code provided since it Actor Critic methods combine both the capabilities of policy based and value based methods, and since the provided starter code serves as a clean framework to start with.
  • The hyperparameters have been updated to reflect the advice given by the reviwer (for learning rate), which seemed to produce better results than previously. The remaining parameters have been kept as is:
    • learning rate for critic: 0.001
    • learning rate for actor: 0.0001
    • gamma: .99
    • tau: .01
    • exploration sigma: 0.2
    • exploration theta: 0.15
    • exploration mu: 0
    • replay memory buffer size: 100000
    • replay batch size: 64
  • I have tried some variations on the architecture, however, after recieving feedback, I've applied the values suggested by the reviwer, and they seemed to improve the learning process. The structure is as follows:
    • Actor: Three fully connected layers with 128, 256, and 300 respectively (At first I trid 512 for the third layer, but 300 worked better).
    • Critic: 2 layers for the state pathway, and 2 for the action pathway, each is 32 and 64 respectively. I tried doubling the number of units on each layer, but that did not introduce improvement in the model.

The second task was not as consistent, but I have been observing the final values of each episode, and while it is not staying exactly at the target point, it is not crashing, and it's hovering at least close to one of the desired axis.
  • Most of the time, there was an aha moment, where suddenly the agent would seem to get it, and start performing better from then on.
  • Usually the last runs are performing better than earlier runs. If you look at the charts right after the execution of each task, you'll notice that the moving average gets better and stays mostly up for the remainder of the task (In the hovering task, it still dips sometimes, but it is still hovering, and not crashing). I've also plotted the final heights for each episode, and you can see that it is doing quite well.

Question 4: Briefly summarize your experience working on this project. You can use the following prompts for ideas.

  • What was the hardest part of the project? (e.g. getting started, plotting, specifying the task, etc.)
  • Did you find anything interesting in how the quadcopter or your agent behaved?

Answer:

  • The hardest part of the project for me was visualizing what the quadcopter is doing based on the information given, since it's moving in 3D space with different velocities, and all I'm seeing is numbers. It took me a while to make sense of these numbers. Additionally, I struggled at first with trying to get the agent to make ANY improvement with the initial reward structure. I only managed to get it to improve once I modified the rewrad, and noticed that if you have a well-designed learning agent with a proper architecture, it will not perform any better than a very simple agent unless its given correct rewards that guide it towards learning. With more computing time, and better visualization, I believe this model can be improved much more in order to make it fly to specific points effectively.
